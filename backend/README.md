# Explainable Multi-Class Diabetic Retinopathy Detection

An end-to-end, research-oriented deep learning system that grades diabetic retinopathy
(DR) severity from retinal fundus photographs into 5 clinical classes, with built-in
**Grad-CAM explainability** and a deployable **Streamlit** demo app.

| | |
|---|---|
| **Task** | 5-class image classification (No DR, Mild, Moderate, Severe, Proliferative DR) |
| **Dataset** | [EyePACS](https://www.kaggle.com/c/diabetic-retinopathy-detection) (Kaggle) |
| **Model** | EfficientNetB0, ImageNet-pretrained, two-phase transfer learning |
| **Explainability** | Grad-CAM + Grad-CAM++ |
| **Deployment** | Streamlit web app |
| **Optimization** | Dynamic-range TFLite quantization for lightweight inference |

> **Not a medical device.** This is a research/decision-support prototype. See
> [Ethical & Clinical Disclaimer](#ethical--clinical-disclaimer).

## Why EfficientNetB0

EfficientNetB0 was chosen over heavier ImageNet backbones (ResNet50, InceptionV3,
DenseNet121) for three reasons directly tied to this project's requirements:

- **Lightweight**: ~5.3M parameters (~20MB as a `.keras` file, ~4-5MB after TFLite
  dynamic-range quantization) — practical for edge/mobile or low-resource-clinic
  deployment, satisfying the "lightweight inference" requirement.
- **Strong transfer accuracy per FLOP**: compound depth/width/resolution scaling tends to
  transfer well to fundus imagery, where lesions are small and localized.
- **Grad-CAM friendly**: a well-defined final feature map (`top_conv`, 7x7x1280 at
  224x224 input) gives clean, interpretable localization maps.

## Installation

```bash
git clone <this-repo>
cd DR_Detection_Project
python -m venv .venv && source .venv/bin/activate   # optional but recommended
pip install -r requirements.txt
```

Verify the install with the synthetic-data smoke tests (no dataset required):

```bash
pytest tests/test_pipeline.py -v
```

This exercises preprocessing, augmentation, class weighting, both model architectures,
Grad-CAM/Grad-CAM++, and the full metrics suite on randomly-generated images — a useful
sanity check before committing to a long training run on the real dataset.

## Dataset Setup

See **[`dataset/README.md`](dataset/README.md)** for the full Kaggle CLI walkthrough.
Short version:

```bash
kaggle competitions download -c diabetic-retinopathy-detection -p dataset/downloads
cd dataset/downloads && unzip -o "*.zip" && cd ../..

python prepare_dataset.py \
    --source-csv dataset/downloads/trainLabels.csv \
    --image-dir dataset/raw_images \
    --extension .jpeg
```

This produces stratified `dataset/{train,val,test}_labels.csv` manifests that every other
script reads from, preserving EyePACS's natural (heavily imbalanced) class proportions
across all three splits.

## Quickstart

```bash
# 1. Train the EfficientNetB0 model (two-phase: frozen head, then fine-tuning)
python train.py

# 2. (Optional) also train the from-scratch baseline CNN for comparison
python train.py --baseline

# 3. Evaluate on the held-out test set
python evaluate.py --model models/saved_models/efficientnet_b0_best.keras --split test

# 4. Generate the baseline-vs-EfficientNet research comparison
python compare_models.py

# 5. Run inference on a single image (with Grad-CAM)
python inference.py --image path/to/fundus.jpg

# 6. Launch the Streamlit demo
streamlit run streamlit_app/app.py

# 7. (Optional) export a lightweight TFLite model for edge deployment
python optimize_model.py --quantization dynamic --benchmark
```

Or open **`notebooks/DR_Detection_Training_Notebook.ipynb`** for a single narrative
walkthrough covering EDA, preprocessing/augmentation demos, both training phases,
evaluation, Grad-CAM visual analysis, and the research comparison — it calls into exactly
the same modules as the scripts above, so notebook and CLI results are reproducible from
each other.

## Methodology

### 1. Preprocessing (`utils/preprocessing.py`)
Raw fundus images are quality-checked (Laplacian-variance blur detection + dark-pixel
fraction, flagging poor captures), cropped to the circular fundus content (removing black
borders), mildly denoised, and passed through **Ben Graham's local-average-subtraction**
technique (the preprocessing used by the 2015 Kaggle DR competition's winning solution):

```
result = 4 * image - 4 * GaussianBlur(image, sigma) + 128
```

This suppresses inter-clinic illumination variance and makes microaneurysms,
haemorrhages, and exudates substantially more visible before the image is resized to
224x224 and normalized.

### 2. Augmentation (`utils/augmentation.py`)
Implemented as Keras preprocessing layers (`RandomFlip`, `RandomRotation`, `RandomZoom`,
`RandomBrightness`, `RandomContrast`) applied only to the training split, in-graph, for
GPU-accelerated augmentation. Only horizontal flip is used (not vertical) and rotation is
kept moderate, respecting the fact that fundus cameras already center the optic
disc/macula fairly consistently.

### 3. Class Imbalance (`utils/class_imbalance.py`)
EyePACS is heavily skewed (~73% No DR, ~2% Proliferative DR). Two mitigations are
provided: inverse-frequency **class weights** (default, passed to `model.fit`) and
optional **rejection-resampling oversampling** at the `tf.data` level
(`config.USE_OVERSAMPLING`).

### 4. Model (`models/efficientnet_model.py`, `models/baseline_cnn.py`)
EfficientNetB0 (`include_top=False`, ImageNet weights) with a custom head
(GlobalAveragePooling &rarr; BatchNorm &rarr; Dropout &rarr; Dense(256, ReLU, L2-regularized)
&rarr; BatchNorm &rarr; Dropout &rarr; Dense(5, softmax)). The backbone is composed directly
into the functional graph (not wrapped as a nested sub-model) so intermediate layers like
`top_conv` stay on the same computation graph as the final predictions — required for
Grad-CAM's gradient computation to work correctly.

**Two-phase training:**
- *Phase 1 (feature extraction)*: backbone fully frozen, only the head trains, higher LR.
- *Phase 2 (fine-tuning)*: layers from `config.FINE_TUNE_AT_LAYER` onward are unfrozen
  (BatchNorm layers stay frozen throughout to avoid destabilizing pretrained statistics on
  small fine-tuning batch sizes), training continues at a much lower LR.

A separate from-scratch `baseline_cnn.py` (4 conv blocks, no pretraining) is provided as
the research comparison model.

### 5. Training (`train.py`)
Train/val/test are pre-split (stratified) by `prepare_dataset.py`. Callbacks:
`ModelCheckpoint` (best-only), `EarlyStopping` (restores best weights),
`ReduceLROnPlateau`, `CSVLogger`, `TensorBoard`, and a custom **`MacroRecallCallback`**
that computes macro-averaged recall on the validation set each epoch. Checkpointing and
early stopping monitor **macro recall**, not accuracy — for a medical screening tool,
missing a true DR case is a far costlier error than a false alarm, and plain accuracy is
dominated by the majority "No DR" class. Phase 2's checkpoint threshold is seeded from
Phase 1's best score so fine-tuning can't silently overwrite a better Phase-1 checkpoint
with a worse early fine-tuning epoch.

### 6. Evaluation (`evaluate.py`, `utils/metrics.py`)
Accuracy, macro/weighted precision-recall-F1, per-class breakdown, confusion matrix
(raw + normalized, saved as PNG), and the full sklearn classification report, all saved to
`results/`.

### 7. Explainability (`gradcam/gradcam.py`)
Grad-CAM (Selvaraju et al., 2017) and Grad-CAM++ (Chattopadhay et al., 2018, which tends
to localize multiple small lesion sites better than vanilla Grad-CAM) computed against the
`top_conv` feature map. Used in the Streamlit app, the CLI (`inference.py`), and the
notebook's explainability section.

## Evaluation & Results

Running `evaluate.py` and `compare_models.py` after training on the full EyePACS dataset
populates `results/reports/` and `results/figures/` with:

- `metrics_report_test.txt` — accuracy / precision / recall / F1 (macro + weighted) +
  full classification report
- `per_class_metrics_test.csv` — per-class precision/recall/F1/support
- `confusion_matrix_test.png` / `confusion_matrix_test_counts.png`
- `model_comparison.csv` + `model_comparison_bar.png` — baseline CNN vs. EfficientNetB0
- `performance_improvement_analysis.txt` — written interpretation of the comparison

**Result table format** (populated automatically after your training run — the numbers
below are placeholders showing the report's structure, not fabricated results):

| Metric | Baseline CNN | EfficientNetB0 | Improvement |
|---|---|---|---|
| Accuracy | `xx.xx%` | `xx.xx%` | `+x.xx pts` |
| Precision (macro) | `xx.xx%` | `xx.xx%` | `+x.xx pts` |
| Recall (macro) | `xx.xx%` | `xx.xx%` | `+x.xx pts` |
| F1-score (macro) | `xx.xx%` | `xx.xx%` | `+x.xx pts` |
| ROC-AUC (macro, OvR) | `x.xxx` | `x.xxx` | `+x.xxx` |

Every pipeline stage (data loading, both training phases, checkpointing, evaluation,

## Explainability (Grad-CAM)

```bash
python inference.py --image path/to/fundus.jpg
```

prints the prediction, confidence, and per-class probabilities, and saves
`gradcam_output.png` — the original image overlaid with a Grad-CAM heatmap highlighting
the regions that most influenced the predicted grade. The Streamlit app exposes the same
capability interactively, with a toggle between Grad-CAM and Grad-CAM++.

When interpreting heatmaps: attention should concentrate on vascular/lesion structures
(microaneurysms, haemorrhages, exudates, neovascularization), not on the optic disc,
vignette borders, or capture artifacts. Heatmaps dominated by image borders are a signal
the model may be relying on a capture-device shortcut rather than a clinical one — see the
notebook's Explainability Analysis section for a fuller discussion.

## Research Enhancements

- **Baseline vs. EfficientNetB0 comparison** (`compare_models.py`): identical test-set
  evaluation of a from-scratch CNN vs. the transfer-learned model, isolating the effect of
  ImageNet pretraining + two-phase fine-tuning.
- **Performance improvement analysis**: auto-generated written interpretation
  (`results/reports/performance_improvement_analysis.txt`) covering absolute/relative
  metric deltas and parameter-efficiency trade-offs.
- **Explainability analysis**: the notebook's Grad-CAM section is structured for
  qualitative research writeup — localization plausibility, confidence-vs-correctness
  cross-referencing, and failure-case analysis (adjacent- vs. non-adjacent-grade
  confusions).

## Streamlit App

```bash
streamlit run streamlit_app/app.py
```

Features: image upload, predicted grade + confidence score, full 5-class probability
breakdown, Grad-CAM / Grad-CAM++ heatmap overlay (toggle in the sidebar), plain-language
disease description + suggested clinical action per grade, and an image-quality warning
banner for blurry/poorly-captured uploads. The app calls directly into `inference.py`, so
behavior is identical to the CLI path.

## Testing

```bash
pytest tests/test_pipeline.py -v
```

Twelve smoke tests covering preprocessing (shape/range/quality checks), augmentation,
class-weight computation, both model architectures (forward pass + freeze/unfreeze
correctness), Grad-CAM/Grad-CAM++ (heatmap shape, gradient-path correctness, overlay
shape), and the full metrics suite — all on synthetic data, so they run in seconds without
requiring the real dataset. Useful as a pre-flight check before a long training run, and as
CI for any future changes to this codebase.

## Lightweight Inference / Deployment

```bash
python optimize_model.py --quantization dynamic --benchmark
```

Converts the trained `.keras` model to TensorFlow Lite (dynamic-range quantization by
default; `float16` or `none` also available) and benchmarks single-image latency. Validate
a quantized model's per-class recall against the float baseline (`evaluate.py`) before
deploying it — quantization is not guaranteed to be free for a medical-screening use case,
and full int8 quantization is deliberately not the default here since it requires a
representative calibration set and can shift decision boundaries.

## Ethical & Clinical Disclaimer

This project is a **research and educational prototype**. It is **not** a certified
medical device, has not undergone regulatory review (e.g. FDA/CE marking), and must not be
used as the sole basis for diagnosis or treatment. Diabetic retinopathy grading carries
real consequences for patient care; any deployment beyond research/education requires
proper clinical validation, regulatory clearance, and a qualified ophthalmologist in the
loop. See `config.DISCLAIMER`, which is surfaced directly in the Streamlit app.
